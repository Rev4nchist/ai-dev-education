Building MCP Servers

Create your own AI-powered development tools

Learn how to build, deploy, and maintain Model Context Protocol servers for AI-assisted development

Introduction to Building MCP Servers

The Model Context Protocol (MCP) allows AI models to interact with external tools and data sources. While there are many pre-built MCP servers available, creating your own custom server gives you the flexibility to integrate specific tools and services that your AI applications need.

In this tutorial, we'll walk through the process of building a simple MCP server using TypeScript. By the end, you'll have a functional MCP server that can be connected to any MCP host like Claude Desktop, Cursor, or Windsurf.

Customization

Build tools tailored to your specific needs and workflows

Security

Maintain control over your data and how it's accessed

Prerequisites

Before we begin, make sure you have the following installed:

  • Node.js (version 18 or later)
  • npm or yarn package manager
  • Basic knowledge of TypeScript and JavaScript
  • A text editor or IDE (like VS Code)
  • An MCP host for testing (like Claude Desktop, Cursor, or Windsurf)

Building a Simple MCP Server with TypeScript

Let's create a simple MCP server that provides tools for working with weather data and unit conversions. We'll build this step by step, starting with project setup and ending with a fully functional MCP server.

1

Project Setup

First, let's create a new directory for our project and initialize it:

mkdir mcp-weather-server
cd mcp-weather-server
npm init -y

Next, install the required dependencies:

npm install @modelcontextprotocol/sdk
npm install --save-dev typescript @types/node

Create a TypeScript configuration file (tsconfig.json) in the root directory:

{
  "compilerOptions": {
    "target": "ES2022",
    "module": "Node16",
    "moduleResolution": "Node16",
    "outDir": "./build",
    "rootDir": "./src",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules"]
}

Update the package.json file to include build scripts:

{
  "name": "mcp-weather-server",
  "version": "1.0.0",
  "description": "A simple MCP server for weather data and unit conversions",
  "type": "module",
  "main": "build/index.js",
  "scripts": {
    "build": "tsc",
    "start": "node build/index.js",
    "dev": "tsc --watch"
  },
  "keywords": ["mcp", "weather", "typescript"],
  "author": "",
  "license": "MIT",
  "dependencies": {
    "@modelcontextprotocol/sdk": "^0.6.0"
  },
  "devDependencies": {
    "@types/node": "^20.11.24",
    "typescript": "^5.3.3"
  }
}
2

Creating the Basic Server Structure

Create a src directory and add an index.ts file with the basic MCP server setup:

import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "@modelcontextprotocol/sdk/zod.js";

// Create a new MCP server
const server = new Server(
  {
    name: "weather-tools",
    version: "1.0.0",
  },
  {
    capabilities: {
      tools: {},
    },
  }
);

// Start the server with stdio transport
const transport = new StdioServerTransport();
server.listen(transport);

This code creates a basic MCP server named "weather-tools" with stdio transport, which is compatible with most MCP hosts.

Note: The stdio transport is used for local MCP servers running on the same machine as the host. For remote servers, you would use a different transport like HTTP/SSE.

3

Defining Tool Schemas

Let's define schemas for our tools using Zod, which is included in the MCP SDK. Add the following code before the server initialization:

// Schema for the get_weather tool
const getWeatherSchema = z.object({
  location: z.string().describe("The city or location to get weather for"),
  units: z.enum(["metric", "imperial"]).default("metric").describe("Temperature units (metric or imperial)"),
});

// Schema for the convert_temperature tool
const convertTemperatureSchema = z.object({
  temperature: z.number().describe("The temperature value to convert"),
  from: z.enum(["celsius", "fahrenheit", "kelvin"]).describe("The source temperature unit"),
  to: z.enum(["celsius", "fahrenheit", "kelvin"]).describe("The target temperature unit"),
});

These schemas define the parameters that each tool will accept, along with descriptions that will be shown to users in the MCP host.

4

Implementing Tool Functions

Now, let's implement the actual functionality for our tools. Add these functions after the schemas:

// Mock weather data function (in a real app, this would call a weather API)
function getWeatherData(location: string, units: "metric" | "imperial") {
  // This is a mock implementation - in a real app, you would call a weather API
  const mockWeatherData = {
    location,
    temperature: units === "metric" ? 22 : 72,
    units: units === "metric" ? "°C" : "°F",
    conditions: "Partly Cloudy",
    humidity: 65,
    windSpeed: units === "metric" ? 15 : 9.3,
    windUnits: units === "metric" ? "km/h" : "mph",
    timestamp: new Date().toISOString(),
  };
  
  return mockWeatherData;
}

// Temperature conversion function
function convertTemperature(value: number, from: string, to: string) {
  // Convert to Kelvin first (as an intermediate step)
  let kelvin: number;
  
  switch (from) {
    case "celsius":
      kelvin = value + 273.15;
      break;
    case "fahrenheit":
      kelvin = (value - 32) * (5/9) + 273.15;
      break;
    case "kelvin":
      kelvin = value;
      break;
    default:
      throw new Error(`Unsupported source unit: ${from}`);
  }
  
  // Convert from Kelvin to target unit
  let result: number;
  
  switch (to) {
    case "celsius":
      result = kelvin - 273.15;
      break;
    case "fahrenheit":
      result = (kelvin - 273.15) * (9/5) + 32;
      break;
    case "kelvin":
      result = kelvin;
      break;
    default:
      throw new Error(`Unsupported target unit: ${to}`);
  }
  
  return {
    originalValue: value,
    originalUnit: from,
    convertedValue: Math.round(result * 100) / 100, // Round to 2 decimal places
    convertedUnit: to,
  };
}

Tip: In a production MCP server, you would typically connect to real APIs or services. For this tutorial, we're using mock data for simplicity.

5

Registering Tools with the Server

Now, let's register our tools with the MCP server. Add this code before starting the server:

// Register the get_weather tool
server.tool({
  name: "get_weather",
  description: "Get current weather information for a location",
  parameters: getWeatherSchema,
  handler: async ({ location, units }) => {
    return getWeatherData(location, units);
  },
});

// Register the convert_temperature tool
server.tool({
  name: "convert_temperature",
  description: "Convert temperature between different units (Celsius, Fahrenheit, Kelvin)",
  parameters: convertTemperatureSchema,
  handler: async ({ temperature, from, to }) => {
    return convertTemperature(temperature, from, to);
  },
});

This code registers our tools with the MCP server, making them available to any MCP host that connects to our server.

6

Building and Running the Server

Now that we have our complete MCP server code, let's build and run it:

npm run build
npm start

Your MCP server should now be running and ready to accept connections from MCP hosts.

Note: The server will continue running until you stop it (e.g., by pressing Ctrl+C). In a production environment, you might want to run it as a service or in a container.

7

Connecting to an MCP Host

To use your MCP server with an MCP host like Claude Desktop, Cursor, or Windsurf, you'll need to configure the host to connect to your server. The exact steps depend on the host you're using:

For Claude Desktop:

  1. Open Claude Desktop
  2. Go to Settings > MCP
  3. Click "Add Server"
  4. Select "stdio" as the transport type
  5. Enter a name for your server (e.g., "Weather Tools")
  6. For the command, enter the path to your Node.js executable followed by the path to your built server, e.g., node /path/to/mcp-weather-server/build/index.js
  7. Click "Add"

For Cursor:

  1. Open Cursor
  2. Go to Cursor Settings > Features > MCP
  3. Click "+ Add New MCP Server"
  4. Select "stdio" as the Type
  5. Enter a name for your server (e.g., "Weather Tools")
  6. For the Command, enter node /path/to/mcp-weather-server/build/index.js
  7. Click "Add"

For Windsurf:

  1. Open Windsurf
  2. Open Cascade using ⌘L (Mac) or Ctrl+L (Windows/Linux)
  3. Click the hammer icon (🔨) in the toolbar
  4. Click "Configure" to open the MCP configuration file
  5. Add your server to the configuration:
{
  "mcpServers": {
    "weather-tools": {
      "command": "node",
      "args": ["/path/to/mcp-weather-server/build/index.js"]
    }
  }
}
  • Save the file and click "Refresh" (🔄) in the MCP toolbar

    Advanced MCP Server Features

    Once you've mastered the basics of building MCP servers, you can explore more advanced features:

    Error Handling

    Implement robust error handling to provide helpful feedback to users:

    server.tool({
  name: "example_tool",
  description: "Example tool with error handling",
  parameters: z.object({
    input: z.string(),
  }),
  handler: async ({ input }) => {
    try {
      // Tool implementation
      return { result: "Success" };
    } catch (error) {
      // Provide a helpful error message
      throw new Error(`Failed to process input: ${error.message}`);
    }
  },
});

    Authentication and API Keys

    Securely handle API keys and authentication for external services:

    // Get API key from environment variable
const API_KEY = process.env.WEATHER_API_KEY;

// Use the API key in your tool
server.tool({
  name: "get_weather_api",
  description: "Get weather data from a real API",
  parameters: getWeatherSchema,
  handler: async ({ location, units }) => {
    const response = await fetch(
      `https://api.weatherservice.com/data?location=${encodeURIComponent(location)}&units=${units}&apikey=${API_KEY}`
    );
    
    if (!response.ok) {
      throw new Error(`Weather API error: ${response.statusText}`);
    }
    
    return await response.json();
  },
});

    Security Warning: Never hardcode API keys in your code. Use environment variables or secure credential storage solutions.

    Streaming Responses

    For long-running operations, you can use streaming responses:

    server.tool({
  name: "long_running_task",
  description: "A task that takes time to complete",
  parameters: z.object({
    taskId: z.string(),
  }),
  handler: async ({ taskId }, { onProgress }) => {
    // Report progress during execution
    onProgress({ status: "starting", progress: 0 });
    
    // Simulate a long-running task
    await new Promise(resolve => setTimeout(resolve, 1000));
    onProgress({ status: "processing", progress: 50 });
    
    await new Promise(resolve => setTimeout(resolve, 1000));
    onProgress({ status: "finalizing", progress: 90 });
    
    await new Promise(resolve => setTimeout(resolve, 500));
    
    // Return final result
    return { taskId, status: "completed", result: "Task output" };
  },
});

    Best Practices for MCP Server Development

    • Clear Documentation: Provide clear descriptions for your tools and parameters to help users understand how to use them.
    • Input Validation: Use Zod schemas to validate input parameters and provide helpful error messages.
    • Error Handling: Implement robust error handling to gracefully handle failures and provide useful feedback.
    • Security: Protect sensitive information like API keys and credentials. Consider implementing authentication for your MCP server if needed.
    • Testing: Test your MCP server thoroughly with different inputs and edge cases.
    • Monitoring: In production environments, implement logging and monitoring to track usage and detect issues.

    Project Structure for Larger MCP Servers

    For more complex MCP servers, you might want to organize your code into multiple files. Here's a recommended project structure:

    mcp-server/
├── src/
│   ├── index.ts              # Main entry point
│   ├── server.ts             # Server configuration
│   ├── tools/                # Tool implementations
│   │   ├── weather.ts        # Weather-related tools
│   │   ├── conversion.ts     # Conversion tools
│   │   └── index.ts          # Tool exports
│   ├── schemas/              # Zod schemas
│   │   ├── weather.ts        # Weather schemas
│   │   ├── conversion.ts     # Conversion schemas
│   │   └── index.ts          # Schema exports
│   └── utils/                # Utility functions
│       ├── api.ts            # API helpers
│       └── validation.ts     # Validation helpers
├── tests/                    # Test files
├── .env                      # Environment variables
├── .gitignore                # Git ignore file
├── package.json              # Package configuration
├── tsconfig.json             # TypeScript configuration
└── README.md                 # Documentation

    Deploying MCP Servers

    Once you've built and tested your MCP server, you might want to deploy it for broader use. Here are some deployment options:

    Local Deployment

    For personal use or testing, you can run your MCP server locally:

    npm run build
npm start

    Packaging as an npm Package

    To make your MCP server easily installable by others, you can package it as an npm package:

    // Add this to package.json
{
  "bin": {
    "mcp-weather-server": "./build/index.js"
  }
}

    Then users can install and run it with:

    npm install -g mcp-weather-server
mcp-weather-server

    Deploying as a Service

    For more robust deployment, you can run your MCP server as a service using tools like PM2 or systemd.

    # Using PM2
npm install -g pm2
pm2 start build/index.js --name mcp-weather-server

    Containerization

    For even more portability, you can containerize your MCP server using Docker:

    # Dockerfile
FROM node:18-alpine

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .
RUN npm run build

CMD ["node", "build/index.js"]

    Build and run the Docker container:

    docker build -t mcp-weather-server .
docker run -it mcp-weather-server
